# PROTOCOLO SSOT 09: RESOLUÇÃO E PREVENÇÃO DEFINITIVA DE FALHAS CSP (M V3)

## 1. NATUREZA DA FALHA RECORRENTE
Este documento atua como a Fronteira Final (Single Source of Truth) para o bloqueio de desenvolvimento causado por violações da **Content Security Policy (CSP)** do Chrome Extensions Manifest V3, particularmente a proibição de manipuladores de eventos em linha (Inline Event Handlers) no HTML.
O sintoma central deste problema é o bloqueio súbito de execução no Front-End com o seguinte log gravíssimo na camada do DevTools:
`Executing inline event handler violates the following Content Security Policy directive 'script-src 'self' 'wasm-unsafe-eval' 'inline-speculation-rules' http://localhost:* http://127.0.0.1:*'. Either the 'unsafe-inline' keyword, a hash ('sha256-...'), or a nonce ('nonce-...') is required to enable inline execution.`

## 2. ORIGEM FÍSICA: O "FIM" DO UNSAFE-INLINE NO MANIFEST V3
Historicamente, sistemas web e extensões MV2 (Manifest Version 2) permitiam misturar marcação (HTML) com lógica (JavaScript). Uma diretiva de segurança chamada `'unsafe-inline'` permitia o uso flexível de código. 
Com a migração forçada mundial do Google para o **Manifest V3**, focado em privacidade, performance extrema e segurança intransigente, a flag `unsafe-inline` foi **permanentemente banida** do contexto de páginas de extensão (`chrome-extension://`). 

Isso fechou uma das maiores vulnerabilidades da web (XSS), mas impôs uma reconstrução letal para desenvolvedores: **NENHUM CÓDIGO JAVASCRIPT PODE VIVER DENTRO DO HTML.**

## 3. O QUE A INTELIGÊNCIA ARTIFICIAL (E O CÓDIGO) ESTÁ PROIBIDA DE FAZER
Em caráter irrevogável, as seguintes estruturas estão **banidas** da Arquitetura do Upsiden e causarão falha crítica:

### 3.1. Atributos Dicionais de Evento ('on-alguma-coisa')
Qualquer atributo HTML que inicie com 'on' e execute lógica.
❌ Proibido: `<button onclick="iniciarDisparo()">Enviar</button>`
❌ Proibido: `<input type="checkbox" onchange="alternarCRM(this.checked)">`
❌ Proibido: `<form onsubmit="return false">`
❌ Proibido: `<img onload="..." onerror="...">`

### 3.2. Tags de Script Internas
❌ Proibido: 
```html
<script>
  console.log("Isso vai causar erro de CSP");
  let variavel = 1;
</script>
```

### 3.3. Avaliações de String (Eval)
❌ Proibido: `setTimeout("funcaoGlobal()", 1000)` (setTimeout deve receber uma função referenciada, não string).
❌ Proibido: `eval("2 + 2")`
❌ Proibido: `new Function("return dados;")`

## 4. PADRÕES DE ENGENHARIA PERMITIDOS (A SOLUÇÃO)
Toda e qualquer interação com o usuário (cliques, arrastes, preenchimento de formulários, alterações de botões) deve obrigatoriamente ser separada, configurada e injetada através do arquivo genético `.js` (JavaScript externo puro) da respectiva interface.

### 4.1. Acoplamento de Ouvintes (Event Listeners Diretos)
A Abordagem básica consiste em dar uma identidade (`id`) ao elemento e "ouvir" aquela identidade através do script.
```html
<!-- Certo (No HTML): Apenas marcação e Identificadores -->
<input type="checkbox" id="kb-sync-toggle"> Sincronizar Etiquetas WPP
```
```javascript
// Certo (No JS): Acoplamento Seguro.
document.getElementById('kb-sync-toggle').addEventListener('change', (e) => {
   toggleCRMSyncMode(e.target.checked);
});
```

### 4.2. Delegação Profunda de Eventos (Event Delegation)
Como a interface do `Upsiden` trabalha fortemente com injeções dinâmicas de "Templates" inteiros (`c.innerHTML = html;`), o elemento pode não existir no momento do carregamento da página. É ineficiente anexar listeners 1 a 1 para 500 botões que não nasceram.
Para esses casos, a Delegação no "Pai" (document) deve ser utilizada.
```javascript
// Cuidado: Delegação Master (No JS).
document.addEventListener('change', function(evento) {
   // Intercepta qualquer "change" da página e vê se é do elemento procurado.
   if (evento.target && evento.target.id === 'kb-sync-toggle') {
       toggleCRMSyncMode(evento.target.checked);
   }
});
```

### 4.3 Padrão Local Atual da Aplicação Upsiden ('data-click')
Notamos que em painéis antigos do projeto, há um mecanismo próprio rodando no index que monitora qualquer elemento com o atributo `data-click` e faz um `eval()` mascarado ou `window[nome_da_funcao]()`.
Embora engenhoso (e contorne o CSP pois o trigger é um atributo inofensivo monitorado no background), o desenvolvedor IA **esquece disso** em componentes recém-criados ou em checkboxes (que demandam estado) e acaba usando `onchange` ou `onclick` natural por inércia do instinto base de criação web.

A determinação expressa neste Dossiê é que: Para lógicas nativas como Inputs, Selects Cíclicos e Checkboxes complexos que injetamos por `.innerHTML`, nós DEVEMOS recuperar a `tag` recém renderizada na linha de baixo e acoplar um `addEventListener` firme.

## 5. REGRAS DE AUTO-FISCALIZAÇÃO DA IA
Toda vez que uma nova interface, modal ou "card" precisar ser ejetado na tela:
1. **O Agente IA deve rastrear a intenção de injeção:** "Estou construindo uma string de template literal (HTML)..."
2. **O Agente IA deve inspecionar a string:** "Contém as palavras onclick, onchange, onmouseover, onsubmit, oninput?"
3. Se contiver, a injeção falhará imediatamente nas entranhas do Chrome antes do usuário perceber. O agente deve arrancar o atributo na hora, anexar um `id` ou `class` único, e no bloco do JS responsável, executar o acoplamento do `addEventListener`.

## 6. SÍNTESE DO BLOQUEIO PARA O FUTURO
Todo erro que mencionar "CSP directive", "'unsafe-inline'", ou "inline-execution" significa de forma inequívoca que a Injeção Web poluiu as diretrizes do Manifest V3. A solução não é mudar o manifest.json (pois o Chrome reprovará e não subirá a modificação), a solução é **SEMPRE refatorar a marcação XML/HTML separando ESTRUTURA (HTML) de COMPORTAMENTO (JS)**.
