## 2. A TRILOGIA DOS LIMITES DE MEMÓRIA (Service Worker Manifest V3)

### 2.1 A Ilusão do `Timeout/Vazio`
O problema mais grave reportado pela plataforma foi uma falha silenciosa: O sistema passava a exibir `Timeout/Vazio` na interceptação do Motor e a automação do WhatsApp cancelava instantaneamente o envio de mídias pesadas.

A intuição inicial mandava acreditar em falhas no pacote da internet (`fetch`) ou no servidor CDN (Supabase), mas a verdade reside na construção do Chrome V8 JS Engine alocado a extensões. 

No Manifest V3, não existe *Background Page* renderizada de forma perene; existe um *Service Worker*. Esse worker tem uma **pilha de chamadas rigorosamente curta (Call Stack Size) e Garbage Collector hiperativo**.

### 2.2 O Erro Mortal: Manipulação de Array de Buffer In-Memory
A tentativa inicial de conversão gerava um `OOM (Out-of-Memory)` disfarçado de Crash na Stack:
```javascript
// CÓDIGO PROIBIDO (NUNCA REPLICAR)
let binary = '';
const bytes = new Uint8Array(buffer);
// 1º Erro: Loop Gigantesco travando a Thread Única
for (let i = 0; i < bytes.length; i++) binary += String.fromCharCode(bytes[i]); 
const base = btoa(binary);

// CÓDIGO PROIBIDO 2 (NUNCA REPLICAR)
// 2º Erro: Uso do apply() - Estoura instantaneamente a V8 Stack
const chunks = new Array(bytes.length);
chunks.push(String.fromCharCode.apply(null, chunk)); 
```
Essas abordagens nativas em JS para converter um ArrayBuffer (5 Milhões de bytes) operam na Heap JS e matam a extensão. O gatilho que a Extension sofre é silencioso — apenas o IPC devolve "undefined" do nada.

### 2.3 SSOT Padrão-Ouro: C++ Native `FileReader` Binding
Chrome V8 110+ permitiu o acesso direto à classe de `FileReader` nativa do sistema operacional, sem iteradores Javascript, garantindo uma performance `0ms overhead`:

```javascript
// CÓDIGO APTO E OFICIAL PARA MÍDIAS PESADAS NO BACKGROUND.JS
const r = await fetch(url, { headers });
const blob = await r.blob();
const reader = new FileReader();

reader.onloadend = () => {
    responder({ sucesso: true, base64: reader.result }); // C++ bypass completo. V8 nem enxerga.
};
reader.onerror = () => {
    responder({ sucesso: false, erro: 'Falha FileReader' });
};
reader.readAsDataURL(blob); // O browser decifra e coda Base64 na GPU/C++ e só cospe a string pronta.
```
> **Regra Magna 1:** Para qualquer extração de CDN rumo ao Page Sandbox onde trafegar grandes volumes binários se faça necessário, use `FileReader` no Service Worker devolvendo apenas Strings `data:mime;base64,...`. NUNCA trafegue ArrayBuffers via Message Passing (Chrome bloqueia via Port Limiting) e NUNCA itere blocos binários manualmente.

## 3. WA-JS ENGINE: A ARTE DE ENGANAR O WHATSAPP MEDIA CHUNK

O WhatsApp não envia apenas arquivos. Ele analisa hashes e metadados contextuais (`isPtt`, `waveform`, assinatura de bytes OGG). Se uma automação enviar um MP3 puramente via JSON base64, o WhatsApp o encapsula como `Documento (File)` forçando um visual degradado no celular do lide.

### 3.1 A Estrutura Absoluta do "CASO A: Áudio" (Mocking do PTT)
Para que a Upsiden enganasse o sistema e a engine WA-JS enviasse como *Áudio Gravado de Voz (PTT)* de ponta a ponta sem falhas:

1. **Destrinchar o Base64 na Injeção:** Precisamos regenerar um `Blob` local puramente na Window injetada. 
2. **File Masking:** Transformar o Blob em Classe `File` do navegador, não deixando resquícios de "String".
3. **Waveform Forgery:** Gerar random-bytes para "Ondas Sonoras" ou a API do celular descarta a exibição fluida.

```javascript
// PROTOCOLO OFICIAL DE ÁUDIO NO WPP-ENGINE.JS
const isAudio = payload.tipo === 'audio';

if (isAudio) {
    // 1. Simular o Header "Gravando Áudio..." de forma humana
    await InjetorWPP.simularGravacao(chatId, 3000); 

    // 2. Trazer a Data String Nativa e limpar
    const b64Data = b64.includes(',') ? b64.split(',')[1] : b64;
    const blob = InjetorWPP.base64ToBlob(b64Data, 'audio/ogg'); 

    // 3. Forjar o File e forjar Ondas Acústicas
    const file = new File([blob], `ptt.ogg`, { type: 'audio/ogg' });
    const waveform = new Uint8Array(64).map(() => Math.floor(Math.random() * 50) + 10);
    
    // 4. Envio Exclusivo como Audio File Original
    await window.WPP.chat.sendFileMessage(chatId, file, { 
        type: 'audio', 
        isPtt: true, // FLAG ABSOLUTA DE MÁGICA
        waveform: Array.from(waveform) 
    });
}
```
> **Regra Magna 2:** Áudios devem EXCLUSIVAMENTE ser remetidos à porta WPP.chat como classe genérica de File `(File, type, options)`. Entregar `.sendFileMessage(chatId, "data:audio...", options)` para Áudios causa corrupção nas instâncias legadas de React do WhatsApp, resultando em conversões silenciosas para "Documentos". Blob->File é obrigatório.

### 3.2 A Estrutura Absoluta do "CASO B: Mídias Comuns Combinadas"
Mídias comuns (Imagens, Vídeos curtos, Documentos PDF) aceitam "Direct Base64 Injection". No entanto, elas devem unificar layout com as legendas. 

**O Sucesso Fenomenal da Interface Balão:** 
Antes, dividia-se as mensagens (Passo 1 Imagem -> Passo 2 Texto). Agora, os campos são empacotados em um único Hit Request.

```javascript
// O QUE SALVA 40% DO TRÁFEGO NA WA-API:
let waType = mime.includes('image') ? 'image' : (mime.includes('video') ? 'video' : 'document');

const options = { 
    type: waType, 
    filename: payload.nome || 'file',
    caption: textResolved.trim() || undefined // <--- Unificação Absoluta
};

// Injection String Direta para imagens é safe
await window.WPP.chat.sendFileMessage(chatId, b64, options);
```
> **Regra Magna 3:** Nunca enviar um payload de mídia de Imagem que tenha String Vazia no `caption: ""`. Ou seja Undefined ou preenchido. Strings vazias confundem o React Render Layer do WhatsApp para anexos. Se `textResolved` for vazio, passamos o ponteiro `undefined`.

## 4. O CÉREBRO CONTROLADOR DE FLUXO (Automation Controller & Sync)

Tudo isso acima ficaria inerte se os Gatilhos não ativassem em tempo real por causa de assimetria.

### 4.1 O Princípio da Lógica Reativa: Storage Polling e `onChanged`
O painel de Automação Visual deve repassar para o `wpp-engine.js` (rodando na aba hostile do WA) em milissegundos sem forçar um F5 do usuário. O IPC Bridge assumiu essa função com a API `chrome.storage.onChanged`:

```javascript
// no ipc-bridge.js (Content Script Layer)
chrome.storage.onChanged.addListener((changes, area) => {
    if (area === 'local' && changes.ups_wa_auto_reply_config) {
        window.postMessage({ 
            origem: 'CONTENT_SCRIPT_SYNC', 
            config: changes.ups_wa_auto_reply_config.newValue 
        }, '*');
    }
});
```

### 4.2 A Arquitetura Matcher de Multi-Palavra e Gatilhos Inteligentes
O Matcher Regex e o Normalizer devem sempre limpar pontuações e espaçamentos do WhatsApp do cliente, operando em letras minúsculas:

```javascript
// Padronizador do Controlador:
const txtClean = incomingMsgBody.trim().toLowerCase().replace(/[.,!?;:]/g, '');

const keywords = trig.palavra.split(',').map(k => k.trim().toLowerCase());
for (const word of keywords) {
    if (condicao === 'contem') {
        if (txtClean.includes(word)) triggerMatch = true;
    } else {
        if (txtClean === word) triggerMatch = true;
    }
}
```

### 4.3 Fila de Execução Síncrona Respeitosa
O `executeSequentialFollowup` age como o coração do tempo (Cronos). O WhatsApp implementa Bloqueios Temporários (Shadow-Bans Limiters) se você enviar 10 mensagens via JS Injector no mesmo exato 1 milissegundo. O Controller é projetado para operar com Mutex / Sync Async Await obrigatoriamente:

```javascript
// O "FOR" de Passos da Automação (Não usar Promise.all)
for (let i = 0; i < steps.length; i++) {
   const passo = { ...steps[i] };
   // 1. Simulação visual de status "Digitando..."
   // 2. Dispatch Completo da Rota Universal do Motor (Caso A, B ou C)
   await this._esperarDelaySegundos(passo.delay_segundos); 
}
```
> **Regra Magna 4:** Para sequências em Bulk, a promessa de execução V8 deve ser resolvida usando `await` puro em Loops procedurais. NUNCA utilize `Array.map` somado a `Promise.all` em automações WhatsApp, pois isso estressará 100% da fila de chamadas em paralelo e gerará bloqueio de conta ou rejeição do motor `WPPConnect` por encavalamento de requisições API.

## 5. RESUMO EXECUTIVO PARA NOVAS INTEGRAÇÕES (BULK MESSAGING)

Sempre que a equipe ou os arquitetos precisarem gerar novas vertentes de disparo:
1. **Nunca crie múltiplos Listeners Injetores.** Centralize no Universal Dispatcher do `wpp-engine.js`.
2. **CDN / Fetch Requests sempre devem ocorrer no Service Worker** (`ipc-bridge` envia a mensagem; o S.W usa `FileReader` e devolve a DataString). O WA Core injetado está operando numa zona `CSP` suja; tentar um fetch lá falhará e esgotará CORS se o Supabase não estiver com headers frouxos.
3. **Mídias grandes só geram Crash Memória se sofrerem conversões string-by-string.** FileReaders convertem C++ direto sem gargalo Javascript.
4. **Respeite o Tempo de Espera (Humanizer).** Toda camada de API deve respeitar o `await sleep(...)` de Humanização implementado no código de base.
